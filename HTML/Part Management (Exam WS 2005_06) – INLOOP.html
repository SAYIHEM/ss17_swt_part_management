Part Management (Exam WS 2005/06)

Given the class diagram below, implement a simple part management structure for a factory.

In a factory, parts can be components, resources, single components or operating supply items. In our sample factory, there are different part types, defined by the interface IPartType, that can be created by the Factory using the method create(). Note that a Part in this task is not a physical part but an abstraction of it. A factory has a Purchasing and a ReceivingStock department.

A Stock‘s Parts are managed with a java.util.Map<Part,Integer>. Each Part object in the stock is mapped to the number of corresponding physical parts in the stock. The following methods operate on this map:

  • get(): Returns the number of physical parts for a Part or -1 if the part is not in the stock.
  • set(): Sets the number of physical parts in the stock for the given Part. Returns false if the part does not exist or the given number parameter has a negative value. Returns true otherwise.
  • insert(): If the part exists in the stock, this method increases the number of physical parts by the given value, otherwise the Part is inserted into the map with the given number of physical parts.
  • delete(): If the part exists, this method decreases the number of physical parts only if there are enough parts in the stock (true case), otherwise no changes are made (false case). Returns also false if the part does not exist.

The purchasing department wants to observe the stock has to be notified of operations performed on the stock. A ReceivingStock has the two attributes minStockItems and maxStockItems. These attributes define the maximum and minimum number of physical parts for a Part and these limits apply to every Part in the stock separately. All methods that change the number of parts in the stock must emit a signal (notify()) when the number drops below the minimum item count. In case the signal has been emitted, the departments that observe the stock get notified (alarm()). For example, the purchasing buys (buy()) exactly that number of physical parts for the abstract part (whose number just dropped below the minimum value) that is equal to the difference between the maximum number of items (maxStockItems) and the number of remaining parts.

The most important erroneous parameter values must be handled properly. Therefore, the parameter value null should result in a NullPointerException while others like empty Strings, non-positive numbers or negative numbers that do not make sense should cause an IllegalArgumentException.
